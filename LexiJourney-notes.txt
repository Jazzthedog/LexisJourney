should i use the "fable-mode" skill when asking to read the ".claude/handoffs/2026-07-11-160623-lexi-journey-start-development.md"?

Short answer: not by default — use it selectively.

The fable-mode skill makes the model adopt heavier planning/verification habits, which costs tokens and time on every step. Your PROMPTS.md already builds in the discipline that matters most here — each prompt is a single scoped system with a hard Verify gate — so for routine prompts (P0.1 scaffolding, P0.2 fog/grain, P4.x content work) it's overhead without much payoff. The reading of the handoff doc itself definitely doesn't need it; that's just context loading.

Where it does earn its cost is the prompts where a silent bug is expensive or the judgment is subtle:

- P2.1 (scripted kinematic props) — this is the project's riskiest technical bet; the failure mode is props that "compile but feel dead," exactly the kind of thing extra rigor catches early.
- P2.2 (CheckpointSystem) — state snapshot/restore bugs are classic silent corruption that only surfaces chapters later.
- P3.2 (SaveSystem + token scent buff persistence) — save/reload correctness is in the same category.

So the practical rule: run your normal instruction message as-is for most sessions, and for those three prompts prepend /fable-mode (or "use fable mode") before the instruction. Found it. This is a real, well-understood shader bug — not a design/scope issue at all. Here's the mechanism:

Grain.ts's noise function is precision-starved by an unbounded, ever-growing time value.

float random(vec2 co) { return fract(sin(dot(co.xy, vec2(12.9898, 78.233))) * 43758.5453); }
...
float grain = (random(outTexCoord * uTime * 100.0) - 0.5) * 0.12;

uTime is this.game.loop.time / 1000 — total seconds since the Phaser game booted, never reset. A few minutes into a real play session, uTime is in the hundreds, and outTexCoord * uTime * 100 is feeding sin() values in the tens of thousands. The shader declares precision mediump float (required for broad GPU/mobile compatibility), which only gives sin() about 10 bits of precision — at that input magnitude, distinct pixels' inputs round to the same quantized value, so instead of per-pixel static you get whole bands collapsing to identical noise output. That's exactly the banding in your screenshot, and it gets worse the longer the session runs.

Why I never caught this: my verification process reloads the page constantly (every test restarts the scene or navigates fresh), which resets game.loop.time back near zero every time. A real, continuous play session — which is exactly what a stranger playtester does — never gets that reset, so it walks straight into this. This is precisely the kind of thing "played by strangers" catches that my testing methodology structurally couldn't, which is a real gap, not just bad luck.

The fix is small and low-risk: bound the time value before it hits sin(), e.g. mod(uTime, 10.0) instead of raw uTime. Keeps the shader's input magnitude small enough for mediump to handle cleanly, with no visible seam (it's frame-to-frame static noise, not a smooth animation, so a 10-second wraparound is imperceptible).
